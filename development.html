Overview

This development guide captures IB Tech's practical standards and recommended practices for building reliable, maintainable, and secure software. It is intended for engineers, tech leads, and devops practitioners working on IB Tech products and integrations.

Use this document as the single source of truth for day-to-day development: how we branch, review, test, deploy, and monitor code. Where project-specific exceptions are needed, record them in the project's README and SOW.

Development Workflow

We follow an iterative, pull-request-driven workflow that emphasizes small, reviewable changes and continuous integration. Key principles:

  • Make small, focused changes that are easy to review and revert.
  • Keep feature branches short-lived (merge within days
  • Automate checks (linting, tests, security scans) in CI for every push and PR.
  • Document design decisions in PR descriptions and link to relevant tickets.

Typical cycle:

  1. Create a branch from main or the active release branch.
  2. Implement changes and add tests.
  3. Open a pull request with a clear description, screenshots, and test plan.
  4. Address review feedback, squash or rebase as needed, and merge after approvals and passing CI.

Branching Strategy

We use a simple, pragmatic branching model:

BranchPurpose
mainProduction-ready code; always deployable.
release/x.yStabilization branches for scheduled releases.
feature/...New features or experiments; short-lived.
hotfix/...Urgent fixes applied to main and backported.

Branch naming conventions: feature/issue-123-add-login, hotfix/2026-04-06-fix-auth. Keep names descriptive and reference the ticket number when available.

Code Review

Code review is mandatory for all changes. Reviews are a chance to share knowledge, catch bugs, and improve design. Expectations:

  • At least one approving reviewer for small changes; two for larger or high-risk changes.
  • Reviewers check correctness, readability, tests, security implications, and performance considerations.
  • Use the PR description to explain the "why", not just the "what". Include a test plan and rollout notes if applicable.
  • Prefer constructive, actionable feedback. If a change is blocking, explain why and propose alternatives.

Merge rules: passing CI, required approvals, and no unresolved conversations.

Testing Strategy

Testing is layered: unit tests, integration tests, and end-to-end (E2E) tests. Aim for fast unit tests and targeted integration tests; reserve E2E for critical user flows.

Guidelines:

  • Unit tests: fast, deterministic, and isolated. Mock external dependencies.
  • Integration tests: exercise real components (databases, queues) in CI sandbox environments.
  • E2E tests: cover core user journeys; run nightly or on release candidates to limit flakiness.
  • Use test coverage as a guide, not a goal. Focus on meaningful assertions and edge cases.

Example test command (Node.js):

npm test
# or
CI=true npm run test:integration

Continuous Integration and Delivery

CI runs on every push and PR. Typical CI pipeline stages:

  1. Install dependencies and run linters.
  2. Run unit tests and static analysis.
  3. Build artifacts and run integration tests in sandbox.
  4. Publish artifacts to internal registry on merge to main or release branches.

CD: Deployments are automated with gated approvals for production. Use feature flags for gradual rollouts and canary deployments to reduce risk.

Rollback plan: every release must include a documented rollback procedure and a health-check-based automatic rollback where feasible.

Infrastructure as Code

All infrastructure is defined and versioned in code (Terraform, CloudFormation, or Pulumi). Principles:

  • Keep environment parity between staging and production.
  • Use modules to encapsulate reusable patterns (VPCs, databases, IAM roles).
  • Store secrets in a managed secret store and reference them at runtime; never commit secrets to source control.
  • Review infrastructure changes via PRs and run plan/apply in CI with approvals for production changes.

Example Terraform workflow:

terraform init
terraform plan -out=plan.tfplan
# Review plan in CI
terraform apply plan.tfplan

Security Practices

Security is integrated into the development lifecycle. Key practices:

  • Perform threat modeling for new features that handle sensitive data.
  • Run automated SAST and dependency vulnerability scans in CI.
  • Enforce least privilege for service accounts and IAM roles.
  • Require HTTPS/TLS for all external endpoints and validate certificates.
  • Rotate credentials and keys regularly; use short-lived tokens where possible.

Incident response: maintain an incident playbook with roles, communication channels, and post-incident review steps.

Observability and Monitoring

Production systems must be observable. We track metrics, logs, and traces to detect and diagnose issues quickly.

  • Metrics: instrument key business and system metrics (error rate, latency, throughput).
  • Logging: structured logs with request IDs and context; centralize logs in a searchable store.
  • Tracing: propagate trace IDs across services to follow request flows.
  • Alerts: configure actionable alerts with runbooks and avoid noisy thresholds.

Example alert: "P95 latency > 1s for 5 minutes" with a runbook that includes steps to identify recent deploys, check downstream services, and roll back if necessary.

Release Management

Releases follow a predictable cadence and clear ownership. For each release:

  • Create a release note summarizing changes, migration steps, and known issues.
  • Tag the release in source control and publish artifacts to the registry.
  • Coordinate with stakeholders for database migrations and breaking changes.
  • Use feature flags and staged rollouts to minimize user impact.
-release: run smoke tests, monitor key metrics, and hold a short post-mortem for any incidents.

Onboarding & Contribution

New engineers should be productive quickly. Onboarding checklist:

  1. Complete access requests (repo, CI, cloud console, secret manager).
  2. Run the local development environment and a sample integration.
  3. Read the architecture overview and the project's README and CONTRIBUTING.md.
  4. Pair with a buddy for the first feature or bug fix.

Contribution guidelines:

  • Follow the project's coding standards and linting rules.
  • Write tests for new behavior and update documentation.
  • Keep PRs focused and include a clear description and test plan.

Troubleshooting & Best Practices

When debugging production issues, follow a structured approach:

  1. Gather context: request IDs, timestamps, recent deploys, and error traces.
  2. Reproduce in staging if possible; avoid making untested changes in production.
  3. Use feature flags to isolate or disable problematic functionality.
  4. Document the root cause and remediation steps in the incident tracker.

Best practices summary:

  • Automate repetitive tasks and keep manual steps documented.
  • Prefer observability and small releases over large, risky deployments.
  • Invest in developer experience: fast tests, clear logs, and reliable local environments.
