IB Tech — Development FAQ

Answers to common development, deployment, and operational questions
How do I get started with a local development environment?
Follow the project's README: clone the repo, install dependencies, and run the local bootstrap script. Typical steps:
  • Install Node.js (LTS) and Docker for services.
  • Run cp .env.example .env and set required variables.
  • Start services with docker-compose up or npm run dev.
  • Run migrations and seed data if applicable: npm run migrate.
If you hit issues, include logs and the output of npm run doctor when contacting devops.
What is the branching and release strategy?
IB Tech uses a pragmatic model: main is always deployable. Feature work happens on short-lived feature/* branches. Release branches (release/x.y) are used for stabilization. Hotfixes use hotfix/* and are merged back to main and release branches. Tag releases and publish artifacts; use feature flags for risky changes.
What are the CI requirements for pull requests?
Every PR must pass automated, unit tests, and security scans. Required approvals depend on risk: one reviewer for small fixes, two for major changes. Include a clear description, test plan, and link to related tickets. Merge only after CI passes and required approvals are present.
How should I write and run tests?
Follow the testing pyramid: fast unit tests, targeted integration tests, and limited E2E tests for critical flows. Use mocks for external services in unit tests. Run tests locally with and integration tests in CI against sandbox environments. Keep tests deterministic and avoid flaky assertions.
How do I authenticate API requests in development and production?
Use scoped API keys for server-to-server calls and OAuth 2.0 for user-authorized flows. Store secrets in environment variables or a secrets manager (Vault, AWS Secrets Manager). Never commit keys to source control. Rotate keys regularly and use separate keys for sandbox
What is the recommended approach for database migrations?
Use a migration tool (Flyway, Liquibase, or framework-native migrations). Keep migrations small and reversible when possible. Run migrations in CI against a staging environment and include a rollback plan. Coordinate schema changes that require application changes with feature flags and phased
How do we handle secrets and configuration?
Store secrets in a managed secret store and inject them at runtime. Use environment (sandbox, staging, production). For local development, use a secure local secret store or a developer-only vault. Audit access to secrets and enforce least privilege.
What are our observability expectations? Instrument services with metrics (latency, error rate, throughput), structured logs with request IDs, and distributed tracing. Configure actionable alerts with runbooks. Use dashboards to track SLOs and monitor for regressions after deploys. Correlate logs, traces faster root cause analysis.
How do I test and secure webhooks?
Use a tunneling tool (ngrok) for local testing. Verify webhook signatures using the shared secret and reject requests with invalid signatures. Implement idempotency for webhook handlers and log request IDs for troubleshooting. Retry logic should be exponential with jitter.
What is the recommended CI/CD pipeline for deployments?
CI should run linting, tests, and security scans. On merge to main, build artifacts and run integration tests in staging. CD should support gated approvals for production, canary or blue/green deployments, and automatic rollback based on health checks. Use feature flags for gradual exposure.
How do we manage dependencies and vulnerabilities?
Keep dependencies up to date and run scans in CI. Use tools like Dependabot or Renovate to open PRs for upgrades. For critical vulnerabilities, follow the incident process: patch, test, and deploy with priority. Maintain a policy for acceptable risk and required
What are the performance and load testing guidelines?
Define performance goals (P95, P99 latency, throughput). Run load tests in an environment that mirrors production. Use realistic traffic patterns and data volumes. Identify bottlenecks, optimize queries and caching, and validate autoscaling behavior. Include performance tests in release criteria for major features.
How do we handle incident response and postmortems?
Follow the incident playbook: detect, triage, mitigate, and communicate. Assign an incident commander, capture timelines, and preserve evidence. After resolution, run a blameless postmortem documenting root cause, impact, remediation, and action items. Track action items to completion and share learnings.
What are the security best practices developers must follow?
Apply least privilege, validate inputs, use parameterized queries, enforce HTTPS, and run SAST and dependency scans. Perform threat modeling for sensitive features and require security reviews for high-risk changes. Report suspected vulnerabilities to the security team immediately.
How do I request access, resources, or production credentials?
Submit an access request through the internal access management system with justification and duration. Requests should include the minimum privileges required. For production credentials, follow the approval workflow and use short-lived credentials where possible. All access is audited.
What should I include when opening a support ticket?
Provide a concise summary, reproduction steps, environment (sandbox/staging/prod), request or trace IDs, timestamps, and relevant logs or screenshots. Indicate severity and business impact. For urgent incidents, escalate via the on-call channel and include the incident commander.
How do we handle backward-incompatible API changes?
Communicate breaking changes well in advance via release notes and deprecation schedules. Provide migration guides and maintain the old API version for a defined deprecation window. Use semantic versioning and coordinate with integrators to minimize disruption.
Where can I find coding standards and linters configuration?
Coding standards and linter configs live in the repository root (e.g., .eslintrc, prettier.config.js). Follow the style guide in CONTRIBUTING.md. CI enforces linting; fix issues locally before opening a PR to avoid CI
How do I contribute documentation or update runbooks?
Documentation is versioned alongside against the docs or runbook repo with clear changes and examples. Include reviewers who own the area. For urgent runbook updates, follow the emergency update process and notify the on-call team.
Who do I contact for developer support or platform issues?
For developer support, email devops@ibtech.com or open a ticket in the internal support portal. For production incidents, use the on-call rotation and the incident channel. Include project ID, request IDs, and a short summary to speed triage.
If your question isn't listed, contact developer support with details and we'll add a canonical answer.
